# 偽の抽象化より escape hatch

## 原則: 写像できない機能は「存在しないふり」をせず、正直にパススルーする

統一ポートで全プロバイダの機能を吸収しようとすると、必ずどこかのプロバイダにしかない機能（例: 特定モデル専用のパラメータ、特殊なストリーミング形式）に当たる。これを無理に統一モデルへ写像しようとすると、信頼できないシムや実質何もしないダミー実装が生まれ、利用者はそれに気づかずハマる。

## providerOptions でパススルーする

- プロバイダ固有機能は、ポートの共通契約を汚さずに `providerOptions` で渡す。プロバイダ名で名前空間化する（型付き言語では `openai` / `gemini` のようにプロバイダ別に型付けされたフィールドを推奨）
- 名前空間化の意味は一意に定義する: 各アダプタは自分宛ての名前空間だけを読み、それ以外の名前空間は設計上無視する（他プロバイダ向けの値が誤って解釈されたり、エラーとして跳ね返ったりしない）
- これにより、ポートの共通契約は「全プロバイダで保証される最小限」に保たれ、個別機能は明示的に「非ポータブル」な経路を通る

```
# 例（言語非依存の疑似コード）
result = port.generate(input, providerOptions: {
  openai: { reasoning_effort: "high" }  # OpenAI 固有。他アダプタは自分宛て以外の名前空間を無視する
})
```

## escape hatch を使ったコードは非ポータブルだと明示する

- `providerOptions` 経由のコードには、「このプロバイダに依存している」ことをコメント・ドキュメントで明示する
- トレードオフ（差し替え時にこの部分は動作確認が必要）を隠さない。隠すと、後で差し替えた時に初めて発覚する

## capabilities() でプロバイダ非対称性を正直に公開する

- アダプタに `capabilities()` のような問い合わせ手段を持たせ、「このアダプタは何ができて何ができないか」を利用側が事前に確認できるようにする
- できないことを「できる」と偽らない。呼び出し時に初めてエラーになる設計より、事前に非対応が分かる設計の方が安全

## 反面教師: 統一レイヤで抽象化が漏れた事例

LiteLLM のような統一レイヤ製品では、プロバイダ間のエラー体系の差異を吸収しきれず、本来 4xx（クライアント起因）であるべきエラーが 5xx（サーバ起因）として表面化する事例が報告されている。これによりアラートの重要度判定を誤り、インシデント対応の初動が混乱する。

統一レイヤの抽象化は便利だが、境界条件（エラー分類・非対応機能）で漏れると、漏れた箇所ほど気づきにくく実害が大きい。escape hatch を用意し、非対称性を隠さない設計は、この種の事故を防ぐための保険である。

## チェックリスト

- [ ] プロバイダ固有機能を、統一ポートの契約を歪めてまで無理に写像していないか
- [ ] `providerOptions` を使ったコードに「非ポータブル」であることが明示されているか（コメント・ドキュメント）
- [ ] `capabilities()` 等で、プロバイダごとに何ができないかを利用側が事前に確認できるか
- [ ] 「できない」機能を、呼び出し時のエラーで初めて気づく設計にしていないか
- [ ] エラー分類がプロバイダ間で揃うよう、アダプタ側で正準分類にマッピングし直しているか（`references/error-taxonomy.md` 参照）
