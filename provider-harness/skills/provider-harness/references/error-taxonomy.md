# エラー分類の正準セット（error taxonomy）

## 原則: プロバイダ固有エラーは正準分類にマッピングしてから扱う

生のプロバイダエラー（HTTPステータスコード・エラーコード文字列・例外クラス）をそのままアプリ側に伝播させると、プロバイダを差し替えるたびにハンドリングロジックを書き直す羽目になる。全アダプタ共通の正準分類を定義し、**プロバイダ固有エラー → 正準分類へのマッピングはアダプタの責務**と明確にする。ポートやアプリ側のコードは正準分類だけを見てハンドリングを書く。

## 正準分類8種

| 分類 | 意味 | retryable | failoverable | 理由 |
|:---|:---|:---|:---|:---|
| `rate_limited` | 同一プロバイダ内のペース制限超過（429 等） | 可（backoff） | 通常不要 | 待てば同一プロバイダ内で解消するため、まず同一プロバイダでリトライする |
| `quota_exhausted` | 契約上の予算・クォータ超過（429 等だが待っても解消しない） | 不可 | 推奨 | 待っても回復しないためリトライは無意味。別プロバイダへのフェイルオーバーが唯一の回復手段 |
| `auth` | 認証・認可エラー | 不可 | 無意味 | 設定エラーであり、別プロバイダに切り替えても同じ設定ミスを引きずる可能性が高い。人手での設定修正が必要 |
| `invalid_input` | 入力不正 | 不可 | 無意味 | 呼び出し側の入力を直さない限り、どのプロバイダでも同じ結果になる |
| `content_blocked` | コンテンツモデレーションによるブロック | 同一プロバイダ内は不可 | 設計判断で選択可 | プロバイダごとにモデレーションポリシーが異なるため、別プロバイダでは通る可能性がある。意図的なポリシー回避に見えるリスクもあるため、フェイルオーバーするかは明示的な設計判断として選ぶ |
| `timeout` | 応答タイムアウト | 可（N回まで） | N回失敗後に推奨 | 一時的な遅延の可能性があるため即フェイルオーバーせず、限定回数リトライしてから切り替える |
| `transient` | 一時的なサーバ側エラー（500/503 等） | 可（N回まで） | N回失敗後に推奨 | timeout と同様、プロバイダ側の一時的な不調である可能性が高い |
| `unsupported` | 要求した機能をプロバイダが提供していない | 不可 | 状況による | 本筋は `capabilities()`（`references/escape-hatch.md` 参照）で事前に検出し、そもそも呼び出さないこと。呼び出し後に判明した場合は、機能を提供するプロバイダへのフェイルオーバーを検討する |

## マッピングはアダプタの責務

プロバイダ固有のエラー（HTTPステータスコード、エラーコード文字列、例外クラス）から上記8分類への変換は、各アダプタの実装が担う。

- 新しいアダプタを追加するときは、そのプロバイダのエラー体系を正準分類にマッピングする対応表をアダプタ内に持つ
- 同じ HTTP ステータス（例: 429）が `rate_limited` にも `quota_exhausted` にもなり得るプロバイダがある。ステータスコード単体でなく、エラーボディやヘッダの情報も使って判定する
- マッピングの網羅性は契約テスト（`references/contract-testing.md`）の「エラー分類のマッピング」項目で検証する

## チェックリスト

- [ ] 全アダプタが同じ8分類（`rate_limited` / `quota_exhausted` / `auth` / `invalid_input` / `content_blocked` / `timeout` / `transient` / `unsupported`）にエラーをマッピングしているか
- [ ] `quota_exhausted` と `rate_limited` を区別できているか（両方 429 で返るプロバイダがあるため、待って解消するかどうかで判定する）
- [ ] `content_blocked` のフェイルオーバー可否が、暗黙ではなく明示的な設計判断として記録されているか
- [ ] `unsupported` を実行時エラーで検出する前に、`capabilities()` での事前検出を検討したか
- [ ] retryable な分類（`rate_limited` / `timeout` / `transient`）にリトライ上限と backoff 方針が設定されているか
