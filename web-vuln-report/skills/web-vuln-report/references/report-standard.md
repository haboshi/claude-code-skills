# 脆弱性診断報告書 標準構成とプロの示唆（report-standard）

本書は `templates/report.html.j2` が実装する報告書構成の設計意図と、テンプレートを
「機械出力」から「プロの報告書」へ引き上げる加筆（`narrative.json`）の書き方を示す。
参照フレームワークは OWASP Top 10:2025、CVSS 4.0（CVSS-B）、OWASP WSTG、ASVS 5.0.0、CWE、および IPA「安全なウェブ
サイトの作り方」。手動診断（スクリプト非依存）でも同じ構成で組み立てる。

## 1. 業界標準の報告書構成（デファクト）

日本の事業向け脆弱性診断報告書は、経営層と技術者の双方が読める二層構造が定石。

1. **表紙** — 報告書種別、診断対象、診断実施日／発行日、実施者、**認可の根拠**、
   機密区分（CONFIDENTIAL / 取扱注意）。認可の明示は防御的診断の必須要件。
2. **エグゼクティブサマリ** — 経営層向け。総合セキュリティグレード・重大度別件数・最優先所見・
   事業影響を非技術的に。詳細は §2。
3. **診断範囲と手法** — スコープ（対象ホスト・除外・認証要否）、診断方式（非破壊／
   ハイブリッド）、使用ツール、評価基準（OWASP/CVSS/CWE）、**制約事項・免責**。
4. **検出事項の詳細** — 所見ごとの技術詳細。重大度降順に提示（§3 の記述型）。
5. **改善ロードマップ / 優先度** — 即時／短期／中期の段階的是正計画（§4）。本テンプレート
   ではエグゼクティブサマリ内に配置し、経営判断に直結させている。
6. **付録** — 巡回エンドポイント一覧、技術情報、再実行用の中間データ、免責文。

IPA「安全なウェブサイトの作り方」の分類（SQL インジェクション、XSS、CSRF、
セッション管理、アクセス制御、設定不備など）は、本スキルの OWASP Top 10 マッピングと
対応が取れる。事業側への説明では OWASP カテゴリ名に IPA の平易な言い換えを添えると通じやすい。

## 2. エグゼクティブサマリの書き方（経営層向け）

読み手は非技術者。「何がどれだけ危険で、まず何にいくら投じるべきか」を最初の数行で伝える。

- **結論から。** 冒頭 1〜2 文で総合セキュリティグレード（例:「総合評価は B（良好）。
  最優先はソース漏洩につながる 1 件」）を述べ、詳細は後段に回す。
- **グレードの方向を誤読させない。** グレードは **A＝安全 〜 F＝危険**、セキュリティスコアは
  **高いほど安全**。「点数が高い＝悪い」と読ませない（個別 CVSS は逆に高いほど危険）。
- **数値は事業言語に翻訳する。** 「CVSS 7.5」ではなく「認証情報が外部から取得可能で、
  漏洩すれば不正アクセスや情報流出に直結する」と影響で語る。
- **最優先所見を名指しする。** 全 N 件のうち、まず対処すべき 1〜3 件を理由付きで挙げる。
- **傾向を一言で。** 「設定ミス（ヘッダ・Cookie 属性）に集中しており、多くは設定変更で
  低コストに是正可能」等、対処の見通しを与える。
- **グレードとスコアの関係に触れる。** グレードは最悪重大度で上限が決まるため、健全でも
  High が 1 件あれば C 帯になる。必要なら「減点式スコアは高いが High により C に頭打ち」等と
  一言補う（算出仕様は `references/cvss-guide.md` §4）。

## 3. 各所見の記述型（技術詳細セクション）

1 所見を次の型で統一する。テンプレートの `finding` ブロックがこの並びを実装する。

| 要素 | 内容 | テンプレートの対応 |
|---|---|---|
| 所見（タイトル） | 何が問題か（catalog のタイトルを使用） | `f.title` |
| 分類・深刻度 | 重大度／CVSS／OWASP／CWE／確度／検出元 | `meta-row` の各 chip |
| 概要 | 問題の説明（catalog の description） | `f.description` |
| 影響 | 悪用されると事業に何が起きるか | `<dt>影響</dt>` |
| 対象 | 該当 URL・箇所（集約後に列挙） | `<dt>対象</dt>` / `f.affected` |
| 再現／証跡 | どう観測したか（evidence）／CVSS ベクタ | `evidence` / `f.cvss_vector` |
| 推奨対策 | 具体的な是正手順（catalog の remediation） | `remediation` |
| 参考 | 一次情報リンク | `f.references` |

記述の原則:
- **証跡を添える。** 「HSTS が無い」ではなく「応答に Strict-Transport-Security ヘッダが
  存在しない」と観測事実で書く（evidence は非破壊観測の範囲に留める）。
- **確度を明示する。** Medium/Low の所見は「兆候」であることを隠さない。反射＝XSS 成立
  ではない等、確定と推定を区別する（`references/verification-integrity` の精神）。
- **対策は実行可能に。** 設定例・ヘッダ値・段階（まず Report-Only で検証等）まで書く。
- **集約を前提に。** 同一 check_id + title は 1 件に束ね該当箇所を列挙する
  （`scripts/scoring.py` の `merge_findings`）。ページ単位の重複計上をしない。

## 4. プロの「示唆」を出すコツ（narrative の核心）

指摘の列挙で終わらせないことが、テンプレート出力とプロの報告書を分ける。

- **優先度を付ける。** 「深刻度 × 確度 × 悪用容易性 × 資産重要度」で順位化し、上位を明言する。
  全件フラットに並べない。
- **費用対効果を語る。** ヘッダ追加・Cookie 属性付与のような**低コストで広く効く対策**を
  先に束ねて提示し、「まずここから」を作る。設計変更を要するものは中期に回す。
- **段階的ロードマップに落とす。** 是正を時間軸で 3 段階に整理する:
  - **即時（〜1 週）:** 公開機微ファイル除去・危険メソッド無効化など、影響大かつ低工数。
  - **短期（〜1 か月）:** セキュリティヘッダ一式、Cookie 属性、TLS 設定の是正。
  - **中期（〜四半期）:** CSP の nonce 化、依存ライブラリ更新と SCA 導入、CORS 方針見直し。
- **再発防止へ接続する。** 個別修正で終えず、恒久対策（セキュアヘッダの共通化、
  CI への SCA/ヘッダ検査組み込み、リリース前チェックリスト、定期再診断）を示す。
- **限界を正直に書く。** 非破壊・スナップショットである旨、CVE 級判定は外部ツール併用時に
  限る旨を制約事項に明記し、過剰な安全宣言をしない。

## 5. テンプレートとの対応（report.html.j2 ↔ データ）

`scripts/render_report.py` が `scored.json`（採点済み所見）・`crawl.json`（巡回結果）・
`narrative.json`（加筆）を差し込む。各セクションの供給元は次のとおり。

| 報告書セクション | テンプレート位置 | 供給データ |
|---|---|---|
| 表紙 | `section.cover` | `target` / `assessed_date` / `issued_date` / `assessor` / `authorized_by` |
| 1. エグゼクティブサマリ（グレード） | `grade-panel` | `summary.grade` / `grade_rating` / `grade_meaning` / `security_score` / `grade_capped` / `total` / `max_cvss` |
| 1. 総括本文 | `div.narrative` | **`narrative.executive`**（加筆・HTML 可） |
| 1. 重大度別サマリ | `table.matrix` | `summary.by_severity`（Critical〜Info） |
| 1. 主要な検出事項（上位） | 上位表 | `findings[:8]`（id / severity_ja / cvss_score / title / owasp） |
| 1. 改善ロードマップ / 優先度 | `narrative`（roadmap） | **`narrative.roadmap`**（加筆・HTML 可） |
| 2. 診断範囲と手法 | 範囲表 | `scope.hosts` / `pages_crawled` / `tools_used` / `authorized_by` |
| 2. 診断手法・制約事項 | 手法／免責 | 静的文（OWASP/CVSS/CWE・非破壊・CVE 級の限界） |
| 3. 検出事項の詳細 | `finding` 反復 | `findings`（description / impact / affected / cvss_vector / evidence / remediation / references / confidence / source） |
| 4. 付録（巡回一覧） | 付録表 | `pages[:40]`（url / status / technologies） |
| 4. 免責 | `disclaimer` | `target` / `tools_used` |

**加筆の入り口は 2 か所だけ:** `narrative.executive`（エグゼクティブ総括）と
`narrative.roadmap`（改善ロードマップ）。`scored.json` を読み、事業リスク文脈・優先度・
段階的計画をこの 2 フィールドに HTML で書く。形式は `SKILL.md` Phase 5 の JSON 例に従う
（`executive` は `<p>`、`roadmap` は `<ol><li>…</li></ol>` が扱いやすい）。数値・件数・
所見一覧はテンプレートが `scored.json` から自動描画するので、narrative では**解釈と示唆**に
集中し、機械が出す事実の焼き直しをしない。

## 6. 品質チェック（提出前）

- [ ] 表紙に認可の根拠が入っているか（未認可の診断は成果物にしない）。
- [ ] エグゼクティブサマリが非技術者に伝わるか（専門用語を影響語に翻訳したか）。
- [ ] 最優先所見が名指しされ、理由が示されているか。
- [ ] 各所見に影響・証跡・具体的対策・確度が揃っているか。
- [ ] ロードマップが即時／短期／中期に整理され、費用対効果に触れているか。
- [ ] 制約事項に非破壊・スナップショット・CVE 級判定の限界が明記されているか。
- [ ] 同一所見が重複計上されていないか（集約されているか）。
