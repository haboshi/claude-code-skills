# harvest-protocol（改善ループの規律）

## 原則: スキルの自動書き換えはしない。提案するのは常に人間が判断できる diff

`/provider-harvest` はスキルを勝手に書き換えない。実装から得られた知見を「提案 diff」として出すだけで、採否は必ず人間が判断する。この境界を崩すと、検証されていない改善がスキルに静かに混入し、後から誰が何を根拠に追加したのか追えなくなる。

## Maker / Checker 分離

改善提案は2段階のレーンに分ける。

- **Maker（抽出役）**: fresh context で動く。改善履歴やこれまでの harvest 結果は渡さない（blind）。渡すのは実装抜粋と現行スキルの型だけ。過去の提案に引きずられた「なんとなく良さそうな一般化」を防ぐ
- **Checker（検証役）**: Maker とは別レーンで動き、提案を敵対的に反証する。「プロジェクトの実コードと現行スキルの型の、具体的な乖離」を引用できない提案は却下する

採用バーは「具体的な乖離を引用できるか」。抽象的な「もっとこうした方が良い」は通さない。これにより、改善提案が際限なく積み重なってスキルが肥大化・ノイズ化するのを防ぐ。

## 採用時の手順

1. 提案 diff を別タスクとして起票する（harvest 実行と同一コンテキストで直接コミットしない）
2. reviewer の承認を得る
3. コミットする。semver は変更の性質で決める:
   - 事実の修正（誤った記述の訂正） → patch
   - 新しい型・パターンの追加 → minor
   - 既存原則の破壊的変更 → major

## 還流を人間の習慣に頼らない: 3点の強制機構

harvest は「気が向いたらやる」では定着しない。以下の3点で強制する。

1. **SessionEnd 検知フック** — 実装セッション終了時に harvest 未実行を検知する
2. **SessionStart 督促** — 次回セッション開始時に、未処理の harvest 候補があれば提示する
3. **ドメインスキル最終ステップ** — 各ドメインスキル（`provider-image-gen` 等）のワークフローの最終ステップとして harvest 実行を組み込み、実装フローの一部にする

この3点がなければ、harvest は「やった方が良いが誰もやらない」タスクになりがちである。仕組みで強制することで、複利的にスキルの型が実装から学習していく状態を維持する。

## `--dry-run` の位置づけ

`/provider-harvest` の通常実行は、提案 diff を生成した上で pending キュー（フックが書き溜めた未処理の harvest 候補）を消化する。いずれの実行でもスキルファイル自体は書き換えない — 生成されるのは提案 diff のみで、コミットは別タスクで人間が判断する（「採用時の手順」参照）。

`--dry-run` を付けると提案 diff の生成のみを行い、pending キューは消化せず保持したままにする。キューを消費せずに提案内容だけを試したい、あるいは動作確認だけしたいときに使う。実装完了直後の Step 5（本スキル SKILL.md 参照）では通常実行を使い、キューをその都度消化する。

## チェックリスト

- [ ] harvest はスキルを直接書き換えず、提案 diff を出すだけになっているか
- [ ] Maker は改善履歴を渡されない fresh context で動いているか（blind であることを確認したか）
- [ ] Checker が却下した提案に「プロジェクト実コードとの具体的な乖離」が引用できていたか
- [ ] 採用した diff は別タスクで起票し、reviewer 承認を経てからコミットしているか
- [ ] semver（patch / minor / major）の判断が変更の性質と一致しているか
- [ ] 実装完了時に `/provider-harvest --dry-run` の実行そのものをスキップしていないか
