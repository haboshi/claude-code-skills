**Goal:** team-dev コマンドに Fable 5 フォールバック機構とエバリエーターゲート（Gate P: 企画段階 / Gate F: 完了報告前、Claude + Codex 二軸）を追加する。

**Architecture:** 対象は Markdown 指示ファイル2つ（コードではない）。`~/.claude/commands/team-dev.md` に (1) Preflight マイクロプローブ + フォールバック節、(2) spawn 自己報告、(3) エバリエーターゲート新セクションと既存ゲートへの接続を Edit で外科的に挿入し、`~/.claude/agents/team-guardian.md` に役割分担の注記を1行追加する。スペックは `docs/superpowers/specs/2026-07-04-team-dev-v2-design.md`（承認済み）。

**Tech Stack:** Markdown（Claude Code カスタムコマンド/エージェント定義）、git（`~/.claude` は独立した git リポジトリ）

## Global Constraints

- 全出力・コミットメッセージは日本語。コミットに Claude への言及・Co-Authored-By・絵文字・AI フッターを付けない
- `~/.claude` リポジトリには無関係の dirty ファイルがあるため、`git add` は必ず対象ファイルパスを明示指定する（`git add -A` 禁止）
- Edit の old_string は本計画に記載の原文を一字一句そのまま使う（行頭の番号・記号・全角/半角を含む）。Edit 前に対象ファイルを必ず Read する
- 依頼範囲外の箇所（既存の文言・構成）を「ついでに」修正しない（surgical-changes 準拠）
- 降格表は fable→opus の1段のみ。sonnet への自動降格は書かない
- 検証は各タスク末尾の grep コマンドで機械確認し、期待出力と一致しなければコミットしない

### Fable 5 フォールバック（三段構え）

前提（公式仕様・2026-07-04 調査）: サブエージェントのモデル指定が組織の `availableModels` 許可リスト外の場合、spawn はエラーにならず**親セッションの継承モデルでサイレント実行**される。可用性の事前検証 API は存在しないため、検知は「実行して確認」で行う。

1. **起動時マイクロプローブ**（Preflight ステップ6）: プローブの自己報告でセッションのモデル表を確定する。不可用なら降格表を適用
2. **spawn 後セルフレポート検証**（常時）: 全メンバーは初回報告に稼働モデルIDを含める（Spawn チェックリスト必須項目）。Lead は指定モデルとの不一致を検出したら、該当メンバーに shutdown を送り **opus を明示指定して再 spawn** する。未統合の worktree 変更がある場合は、Lead が先に統合または破棄を判断してから再 spawn する（品質差し戻しラダー2回目と同じ手順）。プローブ通過後のセッション途中の上限到達を拾う保険層
3. **settings による自動チェーン（推奨）**: settings.json に `"fallbackModel": ["opus"]` の設定を推奨（公式機構 — overloaded / unavailable / サーバーエラー時に自動試行）。また Fable 5 はセキュリティ/バイオ領域のコンテンツ classifier により Opus へ自動降格することがある（transcript に通知が出る）。Lead がこの降格通知を観測した場合も、当該セッションでは降格表を適用する

**降格表: fable → opus の1段のみ。** 降格表の適用中は、品質優先モード・タスク属性ルーティング・品質差し戻しラダーの fable 指定をすべて opus に読み替える。opus も不可用の場合はユーザーへエスカレーションする（sonnet への自動降格は品質モードの意図を壊すため行わない）。

## エバリエーターゲート（Gate P / Gate F）

設計原則: 評価者は Maker と分離した fresh context で起動する（**blind 評価** — 成果物と rubric だけを見せ、Lead の検討過程・弁明・改善履歴は渡さない）。決定論チェック → LLM 評価のカスケード順を守る（決定論で落ちるものを LLM に回さない）。**指摘の生成は二軸（Claude + Codex）、採否と差し戻し判定は Lead**（生成と承認の分離）。設計背景は claude-code-skills リポジトリの `docs/harness-research/` および `docs/superpowers/specs/2026-07-04-team-dev-v2-design.md` を参照。

### verdict 契約（両ゲート・両軸共通）

評価者には以下の構造化 JSON で回答させる:

```json
{
  "status": "pass | revise | escalate",
  "defects": [{ "severity": "CRITICAL|HIGH|MEDIUM", "point": "指摘", "basis": "根拠" }],
  "next_directive": "revise 時に次に直すべきことを具体的に1つ"
}
```

### 共通ループ規律

- 二軸一致の指摘は原則採用（高信頼）。単独指摘は Lead が根拠を見て採否判定
- 同一ゲートの revise は **2回まで。3回目は AskUserQuestion でユーザーへエスカレーション**
- **no-progress 検知**: 前回と同一指摘が再出現したら回数を待たず即エスカレーション（空回りループの防止）
- Codex CLI 未導入・出力パース失敗時は Claude 単軸に自動縮退し、縮退した事実を最終報告に明記する
- 各ゲートの verdict（二軸の一致/不一致を含む）は最終報告の「エバリエーター判定サマリー」に記録する

### Gate P: 企画エバリエーターゲート（Wave 0→1）

mission-brief.md 完成後・Wave 1 実装者の spawn 前に必ず実施する。手戻りコストが最も高い上流での考慮漏れを止める。

1. **Claude 軸**: Task(subagent_type: oh-my-claudecode:critic, model: opus) を fresh context で起動。渡すのは **mission-brief.md とユーザーの元要件のみ**（Lead の検討過程は渡さない）
2. **Codex 軸**（Claude 軸と並列実行。未導入時は `|| true` で自動スキップ）:
   ```bash
   codex exec -s read-only -o /tmp/codex-gate-p.json \
     "mission-brief.md と以下の元要件を読み、rubric の6観点で考慮漏れを判定し、verdict 契約の JSON のみを出力せよ。<元要件とrubric全文をここに展開>" || true
   ```
3. **rubric（考慮漏れの6観点）** — 両軸のプロンプトに全文を含める:
   - 元要件のカバレッジ（全要求がタスク一覧に落ちているか）
   - 受け入れ基準の検証可能性（各基準が「何を実行すれば証明されるか」に落ちるか）
   - インターフェース契約の完全性（エラー形・境界値・共有型を含むか）
   - リスク・エッジケースの見落とし
   - ファイル境界の整合（並列実装の競合の芽がないか）
   - 過剰スコープ（YAGNI — 要求にない作り込みの混入）
4. revise の場合: Lead が mission-brief.md を修正（複雑なら oh-my-claudecode:analyst に再委譲）して再ゲート
5. pass して初めて Wave 1 の実装者を spawn する

### Gate F: 完了エバリエーターゲート（最終報告前）

Wave 3（最終統合検証）で、ユーザーへの最終報告の**前**に実施する二段カスケード。

**前段（決定論・必須）**: 統合マージ後の build 成功 / `npx tsc --noEmit` 通過 / 全テスト通過 / Guardian・Tester の CRITICAL・HIGH 指摘 0 件（= Gate 1→2 と Gate 2→3 の通過）。前段を通過して初めて後段へ進む。

**後段（LLM 二軸・blind）**:
1. **Claude 軸**: Task(subagent_type: oh-my-claudecode:verifier) を fresh context で起動。渡すのは **mission-brief.md の受け入れ基準 + 統合後 diff + テスト実行結果の証拠のみ**。判定内容: 各受け入れ基準が**証拠付きで**達成されているか / 要件の抜け漏れ / スコープ逸脱
2. **Codex 軸**（並列実行。未導入時は自動スキップ）— 統合後の最終状態に対して:
   ```bash
   codex exec review -o /tmp/codex-gate-f.txt \
     "mission-brief.md の受け入れ基準に対する抜け漏れ・スコープ逸脱と、コード品質の最終確認を行い、verdict 契約の JSON を含めて出力せよ" || true
   ```
   > Guardian の Wave 2 の Codex レビューは「実装中の中間検査」、Gate F は「統合後の最終検査」として役割を分ける
3. fail 時: Lead が採否判定 → **品質差し戻しラダー**に接続（1回目: 同一実装者へ修正指示、2回目: モデル1段昇格で再 spawn）。修正後は前段（決定論）から再実行して再ゲート
4. pass 後にユーザーへ最終報告（エバリエーター判定サマリーを含める）
