# durable / volatile 分離

## 原則: スキル・コードに焼き込んでよい情報と、焼き込むと必ず腐る情報を区別する

外部プロバイダ統合のドキュメントやコードは、放置すると陳腐化した情報（廃止されたモデルID、変わった料金、古いAPIシグネチャ）が残り続け、それを信じた実装が静かに壊れる。durable と volatile を最初に切り分けておくと、どこを見直せば良いかが機械的に分かる。

## durable（スキル・コードに焼き込んでよい）

- アーキテクチャの型（ポート設計・アダプタ分離・escape hatch の使い方）
- チームの意思決定（例: 「structured output は必須」「タイムアウトはユースケースごとに明示設定する」）
- セキュリティ不変条件（例: 「ログに API キー・レスポンス内の個人情報を出力しない」）
- エラー分類の設計（レート制限・認証失敗・入力不正をどう区別してハンドリングするか）

これらは「12ヶ月後も正しいか？」という問いにほぼ「はい」と答えられる情報である。

## volatile（焼き込み禁止）

- モデルID・モデルバージョン
- SDK のバージョン番号
- 料金・レート制限の具体的な数値
- 正確な API シグネチャ（パラメータ名・型）
- モデルの能力表（何ができて何ができないか）

これらは数週間〜数ヶ月で変わる。スキルの本文やコードのコメントに直接書くと、いずれ嘘の情報源になる。

## 妥協形: `last_verified` 付きカタログ参照ファイル

volatile な情報を完全に排除できない場合（例: モデルマッピング層は具体モデルIDを持たざるを得ない）は、以下の形で「いつ検証したか」を明示する。

```yaml
# model-catalog.yaml
general-purpose:
  model_id: "example-model-id"
  last_verified: "2026-06-01"
  stale_days: 90
```

- `stale_days` を超えたら、使用前に Context7 / WebSearch 等で再検証する能動的なゲートを実装に組み込む
- カタログファイルは1箇所に集約し、コード中に散らばせない

## 判定に迷ったら

「12ヶ月後もこの記述は正しいか？」を自問する。

- 「はい」と即答できる → durable。スキル本文に書いてよい
- 「たぶん変わっている」→ volatile。カタログ参照 + `last_verified` に切り出す
- 「分からない」→ 一旦 volatile 側に倒す。durable と誤認して放置するコストの方が高い

## チェックリスト

- [ ] スキル・ドキュメントに具体的なモデルID・バージョン番号・料金を直接書いていないか
- [ ] volatile な情報はすべて `last_verified` 付きのカタログファイルに切り出されているか
- [ ] `stale_days` を超えたカタログエントリを使う前に再検証するゲートが実装にあるか
- [ ] 「アーキテクチャの型」と「具体的な数値・ID」が同じ文章の中で混在していないか（混在は再区分のサイン）
- [ ] durable と判定した記述が、実は特定モデルの挙動に依存した一時的な回避策になっていないか
