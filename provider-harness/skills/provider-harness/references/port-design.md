# ユースケース中心のポート設計

## 原則: 「プロバイダの能力」でなく「アプリの実際の用途」を抽象化する

ポート（インターフェース）を「画像生成API」のようにプロバイダの能力単位で切ると、各プロバイダの機能差がそのままポートの複雑さに転化し、結局どのプロバイダの語彙にも寄らない中途半端な抽象化になる。

代わりに「商品説明文からサムネイル生成」のように、アプリが実際に行いたいことを単位にポートを切る。

- 狭くユースケース特化なほど、プロバイダ間の機能差が漏れにくい（差分を吸収する余地がアダプタ側に残る）
- 広く汎用的なポートほど、最大公約数的な機能しか表現できず、結局 escape hatch だらけになる（`references/escape-hatch.md` 参照）
- ポートが1ユースケースに紐づいていれば、差し替え時の検証範囲も1ユースケース分に収まる

### アンチパターン

```
# NG: プロバイダの能力をそのままポートにした例
interface ImageProvider {
  generateImage(prompt: string, options: AnyProviderOptions): Image
}
```
これは実質「OpenAI or Gemini の呼び出しラッパー」でしかなく、アプリ側は依然としてどのプロバイダの制約に従うべきか意識せざるを得ない。

```
# OK: ユースケース単位のポート
interface ProductThumbnailGenerator {
  generateFromDescription(description: string): ThumbnailResult
}
```
この形なら、アプリ側のコードはユースケースの語彙だけで書け、内部実装（どのプロバイダをどう呼ぶか）は完全にアダプタに閉じ込められる。

## プロンプトはポートでなくアダプタ側に置く

プロンプトはモデル固有の資産である。推論系モデル（reasoning model）は旧世代向けに調整されたプロンプトを与えると、かえって出力が劣化することがある（過剰な chain-of-thought 誘導や few-shot が逆に精度を下げるケースが報告されている）。

- ポート側にプロンプト文字列を持たせると、モデルを差し替えるたびにポートの契約自体を変更する羽目になる
- プロンプトはアダプタの実装詳細として閉じ込め、ポートはあくまで「入力→出力」の契約だけを定義する
- モデルごとに最適なプロンプト戦略が異なることを前提に、アダプタ単位でプロンプトをチューニングできる構造にする

## モデル参照は安定した内部名を経由する

具体的なモデルID（例: `gpt-image-2-2026-03`）を呼び出しコードに直接書かない。"general-purpose" / "long-context" のような、チームが定義した安定的な内部名から具体モデルへのマッピング層を1箇所に集約する。

- モデルの引退・値上げ・後継モデルへの切り替えが、マッピング層の1行変更で済む
- マッピング層自体は `references/durable-vs-volatile.md` でいう volatile な情報（具体モデルID）を保持するので、`last_verified` を付けて管理する

## ストリーミングは最初に決める

ストリーミングが必要なユースケースでは、ポート契約に `stream` を明示的に載せるか、非ストリーミング版と別ポートに分けるかを最初に決める。後付けで `stream` を追加するのはポート契約の破壊的変更になり、既存の呼び出し側・契約テストの両方に影響する。

## 責務分担のまとめ

| 層 | 安定性 | 責務 |
|:---|:---|:---|
| ポート | 安定（durable） | ユースケースの入出力契約 |
| アダプタ | churn を吸収 | プロンプト・モデルマッピング・provider 固有ロジック |

ポートを頻繁に変更しなければならなくなったら、それはポートの切り方がプロバイダ寄りになっているサインである。

## チェックリスト

- [ ] ポート名・メソッド名がプロバイダの機能名（`generateImage` 等）でなく、アプリのユースケース（`generateFromDescription` 等）になっているか
- [ ] プロンプト文字列やモデル固有のチューニングパラメータがポートのシグネチャに漏れていないか
- [ ] モデルIDを呼び出しコードに直接埋め込まず、安定した内部名経由でマッピングしているか
- [ ] このポートを新しいプロバイダに差し替えるとき、契約を変えずに済むか（変える必要があるならポートの切り方を見直す）
- [ ] 1ポートが複数の無関係なユースケースを兼務していないか（兼務は差し替え時の検証範囲を無駄に広げる）
