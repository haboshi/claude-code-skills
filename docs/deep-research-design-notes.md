## 再利用価値のある設計パターン

### 1. モード分岐（時間×深度のプリセット）

| 項目 | compare-lite | deep-full |
|------|-------------|-----------|
| 目標時間 | 5〜8分 | 10〜20分 |
| エージェント数 | 2〜3体 | 3〜5体 |
| ラウンド数 | 2（固定） | 最大3（適応型） |
| WebSearch/エージェント | 最大6回 | 最大10回 |
| WebSearch 合計予算 | 最大18回 | 最大40回 |
| 出力 | 比較マトリクス＋結論 | 構造化レポート |

### 2. 停止条件の明文化（暴走対策として優秀）

- エージェントレベル: ラウンド絶対上限（超過ラウンドは「実行禁止」と明記）、WebSearch 回数上限
- グローバルレベル: 合計予算超過 → 即停止し収集済み情報で統合（打ち切り時も成果を返す）
- 出力なしエージェント → スキップして残りで統合（1体の失敗が全体を巻き込まない）

### 3. 品質ゲート（チェックリスト形式）

- 計画した全トピックが統合されている（または未完了が明記）
- 予算内で完了している
- 鮮度表示が冒頭、Source Count / Source Mix / Self Score / Next Upgrades が末尾
- 比較タスクではマトリクスの観点網羅＋用途別推奨＋反証収集を必須化

### 4. モデルポリシー（固定でなくロール×ポリシーで選択）

検索ロールと統合ロールを分離し、ユーザー指定で上書き可能にする設計。
ただしモデル名のハードコードは実態と乖離しやすく保守コストが高かった（教訓）。

## 実走検証からの教訓（2026-07-04）

- 設計書の精緻さ（986行）と実走品質は別物。R8（実物を走らせる評価）を経ずに「高完成度」と評価していた。
- 予算上限がタイトすぎると、核心の一次情報（公式ドキュメントの廃止告知等）に到達する前に打ち切られる。
  深掘り予算は「ラウンド数」より「有望ソースの全文読了数」に割くべき。
- 勝者（グローバル版）の構成: 155行・サブ質問3〜5分解・有望3〜5ソースの全文読了・全主張に出典・
  単一ソース主張はフラグ付け。シンプルな規律の徹底が勝因。
