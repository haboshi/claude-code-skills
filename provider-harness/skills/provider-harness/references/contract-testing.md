# アダプタ契約テストの型

## 原則: 「バージョンが上がっても動くか」に答えるのはテストであり、スキルの文章ではない

どれだけ丁寧にドキュメント化しても、ドキュメントは実行されない限り嘘をつける。「バージョンが上がっても動くか」「プロバイダを差し替えても動くか」という問いに機械的に答えられるのは契約テストだけである。

## 全アダプタが同一の契約テストスイートを通る

ポートが定義する契約（入力の形・出力の形・エラー分類）に対して、全アダプタが同一のテストスイートを実行し、同じ入力セットに対して契約を満たすことを確認する。

- 契約テストはポートの契約に対して書く。特定アダプタの実装詳細に依存させない
- 新しいアダプタを追加したら、既存の契約テストスイートをそのまま流用して合否を判定できる状態を保つ

## モック中心 + 実API の Pin+Verify を最小1本

- 通常の CI ではモック（記録済みレスポンス、VCR的な手法）を使い、高速かつ課金なしで契約を検証する
- 加えて、実 API を叩く契約テストを最小限（例: 1本〜数本）用意し、Pin+Verify のタイミング（`references/pin-and-verify.md`）で手動または低頻度スケジュールで実行する
- 実 API テストを多用しない理由: 課金・レート制限・フレーキーさが増え、CI の信頼性を損なう

## 共通スイートの限界を明記する

共通契約テストスイートが証明できるのは「ポータブルな面（全アダプタ共通の契約）」だけである。`providerOptions` 経由で使う非ポータブルな機能（`references/escape-hatch.md`）は、共通スイートではカバーできない。

- 非ポータブル機能は各アダプタ固有のテストで別途カバーする
- 「共通スイートが緑 = 全機能が保証された」という誤解をチームに広げない。共通スイートは「差し替え可能な部分が保証された」ことしか意味しない

## 契約テストが検査すべき項目例

- **正常系の出力形状** — ポートが約束する型・必須フィールドが揃っているか
- **エラー分類のマッピング** — プロバイダ固有のエラーが正準分類（`references/error-taxonomy.md` 参照）に正しくマッピングされているか
- **タイムアウト挙動** — 設定したタイムアウトで確実に打ち切られるか
- **リトライ対象コードの遵守** — リトライしてよいエラーとしてはいけないエラーが、プロバイダごとの実装で一貫しているか

これらはいずれも「プロバイダを差し替えた瞬間に静かに壊れやすい」項目であり、契約テストで機械的に守る価値が高い。

## チェックリスト

- [ ] 全アダプタが同一の契約テストスイートに対して実行されているか（アダプタごとに別のテストを書いていないか）
- [ ] 契約テストはポートの契約に対して書かれており、特定アダプタの実装詳細（内部フィールド名等）に依存していないか
- [ ] モック中心のテストに加えて、実 API を叩く契約テストが最小1本は用意されているか
- [ ] `providerOptions` 経由の非ポータブル機能が、共通スイートでなく個別テストでカバーされているか
- [ ] 新しいアダプタを追加したとき、共通契約テストスイートをそのまま流用して合否判定できるか
