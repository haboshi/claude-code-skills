# ドメイン別の抽象化の厚さ指針

## 原則: 「一律に同じ厚さで抽象化しない」という判断自体が durable な意思決定

すべてのプロバイダ統合を同じ厚みのポートで抽象化しようとすると、入出力契約が安定しているドメインでは過小に、モデル固有性が強いドメインでは過剰になる。ドメインごとに適切な厚さを選ぶ判断基準自体を、`references/durable-vs-volatile.md` でいう durable な知識としてここに固定する。

## ドメイン別の目安

| ドメイン | 抽象化の厚さ | 理由 |
|:---|:---|:---|
| テキストLLM / STT（音声認識） | 厚い抽象化が効く | 入出力契約（テキスト→テキスト、音声→テキスト）が比較的安定しており、差し替えの価値が高い |
| 画像生成 | 薄く | プロンプトがモデル固有資産で移植不能に近い。ポートは薄くし、アダプタと escape hatch を厚くする |
| リアルタイム音声 | 抽象化より型を持つ | イベント駆動で契約テスト化が最難関。統一ポートを厚くするより「接続ライフサイクルとイベント処理」の型を持つ方が有効 |
| 認証基盤 | 差し替え容易性を最適化目標にしない | セキュリティ不変条件と検証手順の記述場所を優先する。早すぎる抽象化はむしろ危険（認可ロジックの意図しない共通化を招く） |

## 判断の使い方

1. 新しいドメインの統合に着手するとき、まずこの表で近いドメインの目安を確認する
2. 表にないドメインの場合は「入出力契約は安定しているか」「モデル固有性はどれくらい強いか」「イベント駆動か request/response か」を自問し、最も近い行に倣う
3. 判断した厚さと理由は、そのドメインスキル（`provider-image-gen` 等）に記録する。本スキルの表はあくまで初期判断の出発点であり、ドメインスキルが実装知見を蓄積したら harvest（`references/harvest-protocol.md`）で本表の更新候補として還流する

## アンチパターン

- 全ドメインに同一の「プロバイダ抽象化フレームワーク」を適用し、画像生成のプロンプトまで無理に統一ポートに押し込める
- 認証基盤の差し替えやすさを高めるために、テナントごとに異なるべき認可ロジックまで共通化してしまう

いずれも「抽象化は常に厚い方が良い」という誤った一般化から生まれる。ドメインごとに正解の厚さが異なることを前提に設計する。

## チェックリスト

- [ ] このドメインの入出力契約は安定しているか、それともモデル固有性が強いか（表の該当行を確認したか）
- [ ] 画像生成等プロンプト依存の強いドメインで、無理に厚い共通ポートを適用していないか
- [ ] イベント駆動なドメイン（リアルタイム音声等）で、request/response 型の統一ポートを無理に当てはめていないか
- [ ] 認証基盤で、差し替えやすさを優先してセキュリティ不変条件の記述を薄くしていないか
- [ ] 選んだ厚さと理由を、そのドメインスキルに記録し harvest 対象にしているか
