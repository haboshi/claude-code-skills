# Pin+Verify（凍結ではなく固定＋定期検証）

## 原則: 「動いているから触らない」凍結は、いずれ強制的に壊される

外部プロバイダ統合は、放置しても脆弱性修正・モデル引退・API 非推奨化によって強制的に破壊されるタイミングが必ず来る。その時に一気に対応するコストは、定期的に少しずつ検証するコストよりはるかに高い。Pin（固定）は再現性のためであり、思考停止のためではない。

## Pin: 依存をバージョン固定する

- 各プロジェクトで SDK バージョンを lockfile（`package-lock.json` / `poetry.lock` / `uv.lock` 等）で固定する
- モデルIDも `references/durable-vs-volatile.md` のカタログ参照層を通して固定する（直接コードに埋め込まない）
- 固定する目的は「予期しないタイミングでの挙動変化を防ぐ」ことであり、「二度と見直さない」ことではない

## Verify: 依存を上げる時の手順

1. **CHANGELOG / 移行ガイドをレビューする** — 破壊的変更・非推奨項目を洗い出す
2. **契約テストを実行する**（`references/contract-testing.md`）— ポータブルな契約が壊れていないかを機械的に確認する
3. **挙動差分があればアダプタで吸収する** — ポート契約を変えずに済むよう、差分吸収はアダプタの責務にする
4. **catalog の `last_verified` を更新する** — 次回の stale 判定の基準点をリセットする

## なぜ凍結が脆弱か

- セキュリティ脆弱性は依存側の都合で公表される。凍結したままでは、修正版へ追従する経路自体が用意されていない
- モデル・APIの引退はプロバイダ都合で強制される。移行ガイドを読まずに放置すると、引退当日に初めて障害として発覚する
- 「動いているから触らない」を積み重ねると、いざ更新が必要になった時に一度に大量の差分を検証する羽目になり、リスクが高まる

## 運用の目安

- 依存更新のレビューと契約テスト実行は、機能追加のたびに全件やり直す必要はない。カタログの `stale_days` を超えたタイミング、またはセキュリティアドバイザリを検知したタイミングで実施する
- 定期的な verify は「安い」。まとめて強制対応するコストと比較すれば、小さく刻んだ検証の方が常に低リスクである

## チェックリスト

- [ ] SDK バージョンが lockfile で固定されているか（`^` `~` 等のレンジ指定に依存していないか）
- [ ] モデルIDが `references/durable-vs-volatile.md` のカタログ参照層を経由しているか（直書きしていないか）
- [ ] 依存更新時に CHANGELOG / 移行ガイドをレビューする手順がチームに共有されているか
- [ ] 依存を上げた後、契約テストスイート（`references/contract-testing.md`）を実行してから catalog の `last_verified` を更新しているか
- [ ] 「動いているから触らない」判断を、最後に verify した日付を確認せずに下していないか
